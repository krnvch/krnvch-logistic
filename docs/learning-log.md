## 10.4 Страница профиля / настройки — 28.02.2026

### Что сделали

Раньше у пользователя не было никакой возможности управлять своим профилем: имя/фамилия не хранились, пароль можно было сменить только через SQL, а переключение роли требовало админа. Теперь появилась страница `/profile` с самообслуживанием.

### Что появилось

- **Карточка "Личные данные"** — имя, фамилия (опционально), email (только чтение), переключатель роли (оператор/работник)
- **Карточка "Смена пароля"** — свёрнута по умолчанию, раскрывается по клику. Текущий пароль → новый → подтверждение
- **Пункт "Профиль" в меню** — добавлен на обеих страницах (список рейсов и детальная страница рейса)
- **Валидация паролей** — вынесена в чистую функцию `validatePasswordChange()` с 5 unit-тестами

### Что узнал

- **Supabase `updateUser()`** — метод для обновления пользовательских данных. Поле `data: {}` пишет в `raw_user_meta_data` — это JSON-хранилище для произвольных атрибутов пользователя (имя, фамилия, роль). Не нужно создавать отдельную таблицу `profiles` для простых случаев.
- **Верификация пароля через `signInWithPassword()`** — Supabase JS SDK не имеет метода `reauthenticate()`. Поэтому для проверки текущего пароля вызываем `signInWithPassword()` — если пароль неверный, получим ошибку. Если верный — создаётся новая сессия для того же пользователя, что безвредно (auth listener всё обработает).
- **React 19 и `setState` в `useEffect`** — линтер React 19 запрещает вызывать `setState` напрямую внутри `useEffect` (правило `react-hooks/set-state-in-effect`). Это анти-паттерн, потому что вызывает каскадные ре-рендеры. Решение: вместо эффекта для синхронизации состояния — использовать `key` на компоненте. Когда `key` меняется, React полностью перемонтирует компонент с новым начальным состоянием. Мы поставили `key={session.user.updated_at}` — после сохранения профиля сессия обновляется, `updated_at` меняется, компонент перемонтируется с актуальными данными.
- **Паттерн "чистая функция + тесты"** — валидацию пароля вынесли в отдельную функцию `validatePasswordChange()`. Это тот же подход, что и с `getOrderStatus()`: чистая функция без зависимостей → легко тестировать. 5 тестов покрывают: слишком короткий пароль, несовпадение, валидный случай, приоритет ошибок, граничный случай (ровно 6 символов).
- **Collapsible-секция без библиотеки** — секция смены пароля свёрнута по умолчанию. Вместо подключения Accordion-компонента — простой `useState(false)` + условный рендеринг + иконки ChevronDown/ChevronUp. Для одного места это проще, чем тянуть новый UI-примитив.

### Аналогия из дизайна

Страница профиля — это как панель настроек в Figma (аватарка, имя, email). Ты видишь её постоянно, но редко задумываешься, как она работает изнутри. Технически это форма, которая читает данные из Auth-системы и пишет обратно через API. Collapsible-секция — как раскрывающийся блок в свойствах слоя: не занимает место, пока не нужна.

### Технически

- 4 новых файла, 3 изменённых
- 5 новых тестов (всего 15 в проекте)
- Нет новых npm-зависимостей — все UI-компоненты уже были

## 9.1 Мульти-рейсы (параллельные сессии) — 28.02.2026

### Что сделали

Раньше приложение поддерживало только один активный рейс. В реальности операторы ведут 3–5 грузовиков одновременно и записывают всё на бумажках. Теперь всё в приложении.

### Что появилось

- **Страница списка рейсов** (`/`) — таблица со всеми рейсами, фильтры (все/активные/завершённые), сортировка, поиск по названию
- **Детальная страница рейса** (`/shipments/:id`) — тот же трейлер + заказы, но по URL с ID
- **Создание/удаление/переименование/возобновление** рейсов (только оператор)
- **Read-only режим** — завершённые рейсы открываются без кнопок редактирования
- **Быстрый вход** — приложение запоминает последний рейс в localStorage и автоматически перенаправляет туда

### Что узнал

- **Роутинг с параметрами** — `useParams()` достаёт `:id` из URL. Это стандартный паттерн: список → детальная страница. Как в любом интернет-магазине: каталог → карточка товара.
- **Два ортогональных измерения доступа** — `isOperator` (роль) и `isReadOnly` (статус рейса) — разные вещи. Оператор на завершённом рейсе тоже не может редактировать. Это как в Figma: можешь быть Editor, но на заблокированном фрейме всё равно ничего не сделаешь.
- **localStorage для UX** — простой способ запомнить состояние между сессиями. Не надо тащить на сервер. Используется для "быстрого входа" — как "последний открытый документ".
- **React Router state** — можно передать скрытые данные при навигации (`navigate("/", { state: { skipRedirect: true } })`). Это помогло решить проблему: "Все рейсы" в меню должен показать список, а не перенаправить обратно.
- **Миграции БД** — добавили `created_by`, `updated_at`, сделали `name` обязательным. Миграция — это как "безопасный апдейт" базы: сначала добавляем поле, потом заполняем дефолтами, потом делаем NOT NULL. Порядок важен.
- **Realtime подписки** — добавили `shipments` в Supabase Realtime. Теперь если один оператор создаёт рейс, другие видят его в таблице сразу, без обновления страницы.

### Технически

- 7 новых файлов, 10 изменённых, 2 удалённых
- Миграция: `supabase/migrations/20260228120000_multi_shipment.sql`
- Детальный план: `docs/multi-shipment-plan.md`

## 7.1 Branching + Pull Request workflow — 28.02.2026

### Что сделали

Перешли с "push прямо в main" на ветки + PR. Теперь каждая задача делается в отдельной ветке и проходит через Pull Request.

### Что узнал

- **Feature branch** — это черновик. Создаёшь ветку (`git checkout -b feature/название`), работаешь в ней, ломай что хочешь — main не пострадает.
- **Pull Request (PR)** — это запрос "влей мои изменения в main". На PR можно оставить комментарии, попросить ревью, увидеть diff (что изменилось). Это как показать дизайн на ревью перед публикацией.
- **Merge** — когда PR одобрен и проверки прошли, изменения сливаются в main. Как финальный "Approve" в Figma.
- **Branch protection rules** — можно запретить прямой push в main (требовать PR). На бесплатном GitHub это работает только для публичных репо. Но workflow работает и без правил — просто дисциплина.

### Рабочий процесс (как теперь делаем)

```
1. git checkout -b feature/название-задачи    ← создать ветку
2. ... пишем код, коммитим ...
3. git push -u origin feature/название-задачи  ← пуш ветки
4. gh pr create --title "..." --body "..."     ← создать PR
5. ... CI проверяет, ревью ...
6. gh pr merge N --merge                       ← замержить
7. git checkout main && git pull               ← вернуться в main
```

### Аналогия из дизайна

Ветка — это как отдельный frame в Figma, куда ты набрасываешь варианты. Main — финальный макет. PR — момент, когда показываешь вариант команде. Merge — "Approve, го в прод".

## 2.1 CI pipeline (GitHub Actions) — 28.02.2026

### Что сделали

Добавили файл `.github/workflows/ci.yml`. Теперь при каждом PR и пуше в main GitHub автоматически запускает проверки.

### Как работает CI

CI (Continuous Integration) — это автоматический проверяльщик. Файл `ci.yml` — это инструкция для GitHub:

```
"Когда кто-то открывает PR или пушит в main":
1. Установи зависимости (pnpm install)
2. Проверь стиль кода (pnpm lint)
3. Запусти тесты (pnpm test)
4. Собери проект (pnpm build — включает TypeScript проверку)
```

Если любой шаг падает — PR показывает красный крестик. Зелёная галочка = всё ок, можно мержить.

### Что узнал

- **YAML** — формат конфигов (как JSON, но читабельнее). В нём описывается workflow: когда запускать, на какой машине, какие шаги.
- **GitHub Actions** — встроенный CI/CD сервис GitHub. Бесплатно для публичных репо, 2000 мин/мес для приватных.
- **`pnpm install --frozen-lockfile`** — устанавливает зависимости строго по lock-файлу (не обновляя ничего). Это гарантирует, что CI собирает ровно то же, что и ты локально.
- **Первый запуск поймал баги** — на первом CI прогоне нашлись 3 lint-ошибки (2 в shadcn-компонентах, 1 в нашем коде). Починили → второй запуск прошёл зелёным. Именно для этого CI и нужен!

### Практический урок

CI поймал ошибку `setState in effect` — React не рекомендует вызывать `setState` напрямую в `useEffect`. Решение: заменили `useState` на `useRef` — ref не вызывает ре-рендер, и lint доволен.

## 1.1 Unit-тесты (Vitest) — 28.02.2026

### Что сделали

Написали первые тесты для функции `getOrderStatus()` — она вычисляет статус заказа (ожидает / загружен / готов).

### Что такое unit-тест

Unit-тест — маленькая программа, которая проверяет одну функцию. Как чеклист: "при таких входных данных — ожидаю такой результат".

```typescript
// Пример теста:
it("is_done=true → always 'done'", () => {
  const order = makeOrder({ is_done: true, box_count: 10 });
  expect(getOrderStatus(order, 0)).toBe("done");
});
```

**`expect(...).toBe(...)`** — это assertion (утверждение). "Я ожидаю, что результат будет 'done'". Если нет — тест падает красным.

### Что узнал

- **Vitest** — test runner для Vite-проектов. Быстрый, понимает TypeScript и path aliases (`@/`) из коробки.
- **`describe` + `it`** — структура тестов. `describe` группирует, `it` описывает конкретный случай. Читается как предложение: "getOrderStatus returns 'done' when order is marked done".
- **Приоритет правил** — тесты помогли зафиксировать бизнес-логику:
  1. `is_done = true` → всегда "done" (даже если коробки не все на месте)
  2. `placedBoxes >= box_count` → "loaded"
  3. Иначе → "pending"
- **Чистая функция — идеальный кандидат** — `getOrderStatus()` не зависит от базы данных, API или UI. Она берёт данные → возвращает результат. Поэтому тестировать её просто.
- **`pnpm test`** — запустить тесты один раз. **`pnpm test:watch`** — запустить в режиме наблюдения (перезапускает при сохранении файлов).

### Как тесты вписались в CI

Добавили шаг `pnpm test` в CI pipeline. Теперь полный цикл проверок: **lint → test → build**. Если тесты падают — PR не пройдёт.

## Замкнутый цикл: ветки → CI → тесты

Вот как теперь выглядит процесс разработки от начала до конца:

```
Задача → feature branch → код + тесты → push → PR
                                                 ↓
                                            CI автоматически:
                                              lint ✓
                                              test ✓
                                              build ✓
                                                 ↓
                                          Всё зелёное → merge в main
                                                 ↓
                                          Vercel деплоит → прод обновлён
```

Каждое звено цепи усиливает предыдущее:
- **Ветки** дают изоляцию — ломай что хочешь, main не пострадает
- **CI** автоматически проверяет — не забудешь запустить lint/tests
- **Тесты** ловят баги до прода — CI их запускает за тебя
- **PR** даёт видимость — всегда понятно, что менялось и почему
